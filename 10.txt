1. Sourcing the Single Choke Point (src/lib/gameApi.ts)

All client-to-server operations in /home/project/src/lib/gameApi.ts are now systematically routed through a single central execution handler: callServerWithRetry. This completely intercepts and controls all websocket communications:

// Sourced from src/lib/gameApi.ts
export async function callServerWithRetry(server: any, method: string, args: any[] = []): Promise<any> {
  const maxRetries = 3;
  const backoffs = [400, 1000, 2000];
  let attempt = 0;

  while (true) {
    try {
      const res = await server.remoteFunction(method, args);
      // Turn off reconnecting HUD on successful completion
      useAppStore.getState().setReconnecting(false);
      return res;
    } catch (err: any) {
      if (!isTransientError(err) || attempt >= maxRetries) {
        useAppStore.getState().setReconnecting(false);
        throw err;
      }

      attempt++;
      console.warn(`Transient error encountered on call to ${method} [Attempt ${attempt}/${maxRetries}]. Retrying in ${backoffs[attempt - 1]}ms... Error:`, err);

      useAppStore.getState().setReconnecting(true);
      await new Promise(resolve => setTimeout(resolve, backoffs[attempt - 1]));
    }
  }
}

2. Transient vs. Genuine Error Classification
Transient Errors: The utility isTransientError(error) parses the raw exception message and returns true for platform-level socket drops, lock failures, and idle VM reclaim:
"Isolate is disposed" / "disposed"
"$lock timeout" / "timeout"
"Socket closed" / "ws closed" / "connection closed"
WebSocket error code 1006 (dropped connection)
Genuine Game Errors: Returns false for validation errors (such as "INSUFFICIENT_GOLD", "out of range", "NO_ACTIVE_BATTLE"). These are not retried and are surfaced immediately to the player.
3. Stable requestId Re-use Across Retries

For mutating actions (like summon, openBox, or enhanceEquipment), the client-side wrappers generate a unique requestId once at the first invocation:

export async function summon(server: any, costType: "neso" | "scroll", requestId?: string): Promise<any> {
  if (!server) throw new Error("GameServer not connected");
  return await callServerWithRetry(server, "summon", [costType, requestId || generateRequestId()]);
}

Because args is passed to callServerWithRetry, subsequent retries re-use the exact same requestId.
If a transient drop occurs after the server successfully processes the action, the server's lock deduplicator intercepts the retry and returns the cached result. This fully prevents any double-spend, double-pulls, or duplicate equipment drops.
