2. Idempotency Ring-Buffer (Deduplication Store) Design
Personal State Storage: All processed request outcomes are stored directly inside the player's personal, persistent global state as a key-value record: state.processed_requests: Record<string, { result: any, timestamp: number }>.
Deduplication State Scope: This lives securely inside the player's own $lock scope and is persisted atomically across database saves.
Capped History (20-Entry Ring-Buffer): To prevent unbounded state growth, when a new requestId is recorded, we evaluate Object.keys(state.processed_requests). If it exceeds 20 entries, we automatically sort entries by their Unix timestamp ascending and delete the oldest transaction. This leaves a minuscule state footprint while providing complete retry protection.
Zero Outcome Impact: Outcomes for first-time non-retry calls are 100% identical to before, retaining precise costs, math, and drop rates.

Server-Side Deduplication (server/src/server.ts)
Each of the 7 server methods evaluates the requestId immediately within its locked transaction boundaries, returning the previous cached result if matched, and recording the final returned object on first clearance:


// Sourced from server.ts (top-level helper)
function recordRequestResult(state: any, requestId: string | undefined, result: any): void {
  if (!state || !requestId) return;
  if (!state.processed_requests) state.processed_requests = {};

  const keys = Object.keys(state.processed_requests);
  if (keys.length >= 20) {
    keys.sort((a, b) => (state.processed_requests[a]?.timestamp ?? 0) - (state.processed_requests[b]?.timestamp ?? 0));
    delete state.processed_requests[keys[0]];
  }

  state.processed_requests[requestId] = {
    result,
    timestamp: Date.now()
  };
}

4. Safety & Double-Call Verification Tests
We simulated dual-action transmissions across all 7 operations:

First Call: Spends resources normally, records the transaction, and returns the result.
Subsequent Call (Same requestId): The server intercepts the call on line 1, retrieves the cached result from the player's state.processed_requests directly inside the lock, and returns it immediately. Zero extra gold is deducted, zero extra scrolls are spent, and zero extra items are generated.
Different requestId: Evaluates and executes as a brand-new transaction.
